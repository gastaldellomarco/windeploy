# Sei un esperto di architettura database e sicurezza. Progetta lo schema MySQL per

WinDeploy — piattaforma automazione configurazione PC Windows aziendali.
Stack: Laravel 11, PHP 8.3, MySQL 8, sviluppo su XAMPP Windows 11.
 
TABELLE:

1. users: id, nome, email, password(bcrypt), ruolo(enum:admin/tecnico/viewer),
   attivo(bool), created_at, last_login, last_login_ip
2. templates: id, nome, descrizione, user_id(FK), scope(enum:globale/personale),
   configurazione(JSON), created_at, updated_at
3. wizards: id, nome, user_id(FK), template_id(FK nullable), codice_univoco(6char univoco es."WD-7A3F"),
   stato(enum:bozza/pronto/in_esecuzione/completato/errore), configurazione(JSON completo),
   created_at, expires_at(24h), used_at
4. software_library: id, nome, versione, publisher, tipo(enum:winget/exe/msi),
   identificatore(winget id o path), categoria, icona_url, aggiunto_da(FK users),
   attivo(bool), created_at
5. execution_logs: id, wizard_id(FK), pc_nome_originale, pc_nome_nuovo,
   tecnico_user_id(FK), hardware_info(JSON: cpu/ram/disco/windows),
   stato(enum:avviato/in_corso/completato/errore/abortito),
   log_dettagliato(JSON array step con timestamp e esito), started_at, completed_at
6. reports: id, execution_log_id(FK), html_content(LONGTEXT), created_at
 
RICHIESTE:
7. Migration Laravel (PHP) tutte le tabelle in ordine corretto (rispetta FK)
8. Indici per query frequenti (lista wizard per stato, report per data, ecc.)
9. Dati sensibili da NON salvare in chiaro (password wizard) — come gestirli
10. Struttura JSON completa per campo configurazione wizard:
   {nome_pc, utente_admin{username, password_encrypted},
   software_installa[], bloatware_default[], power_plan{tipo, params}, extras{timezone, language, wallpaper_url, wifi, windows_update}}
11. DatabaseSeeder con dati fake per sviluppo locale (1 admin, 2 tecnici, 5 software, 3 wizard)
12. Soft delete: su quali tabelle conviene e perché

Ho tutto il contesto necessario. Ora genero il codice completo per lo schema database WinDeploy.

Ecco lo schema MySQL completo per WinDeploy, con migrations Laravel in ordine corretto, strategie di sicurezza, JSON strutturati e seeder per sviluppo locale.[^1_1]

## Dati Sensibili — Strategia di Cifratura

> ⚠️ **Implicazione di sicurezza critica.** Il campo `utente_admin.password_encrypted` nel JSON `configurazione` dei wizard NON deve mai essere in chiaro nel database.

La strategia consigliata per WinDeploy è **AES-256-GCM via Laravel Crypt**, che è già disponibile senza dipendenze aggiuntive.[^1_1]

### Approccio: Cifratura a livello applicativo (Model Cast)

```php
<?php
// app/Models/Wizard.php

namespace App\Models;

use Illuminate\Database\Eloquent\Model;
use Illuminate\Support\Facades\Crypt;

class Wizard extends Model
{
    protected $fillable = [
        'nome', 'user_id', 'template_id', 'codice_univoco',
        'stato', 'configurazione', 'expires_at', 'used_at',
    ];

    protected $casts = [
        'expires_at' => 'datetime',
        'used_at'    => 'datetime',
    ];

    /**
     * Override getter: decifra la password admin prima di restituire la configurazione.
     * La password viene decifrata SOLO qui, mai salvata in chiaro.
     */
    public function getConfigurazione(): array
    {
        $config = json_decode($this->attributes['configurazione'], true);
        return $config; // La struttura è già con password encrypted nel campo
    }

    /**
     * Cifra la password admin prima del salvataggio.
     * Chiamare questo metodo nel Controller prima di $wizard->save()
     */
    public static function encryptAdminPassword(array $configurazione): array
    {
        if (isset($configurazione['utente_admin']['password'])) {
            $plain = $configurazione['utente_admin']['password'];
            $configurazione['utente_admin']['password_encrypted'] = Crypt::encryptString($plain);
            unset($configurazione['utente_admin']['password']); // rimuovi il campo plain
        }
        return $configurazione;
    }

    /**
     * Decifra la password admin — usato solo dall'endpoint dedicato all'agent.
     * Non includere mai questa operazione nelle API generiche.
     */
    public static function decryptAdminPassword(array $configurazione): string
    {
        return Crypt::decryptString(
            $configurazione['utente_admin']['password_encrypted']
        );
    }
}
```

**Cosa NON fare:**

- ❌ Salvare `password` in chiaro nel JSON (neanche temporaneamente)
- ❌ Restituire `password_encrypted` nelle API di listing (`/api/wizards`)
- ❌ Loggare la configurazione completa — filtra il campo nei log Laravel

**Flusso sicuro:**

1. Frontend invia `password` in chiaro **solo su HTTPS**
2. Il Controller chiama `Wizard::encryptAdminPassword()` prima di `save()`
3. Nel DB viene salvato solo `password_encrypted` (ciphertext AES-256)
4. L'agent riceve la password decifrata **solo via endpoint dedicato protetto**, subito prima dell'uso
5. L'agent usa la password e **non la scrive mai su disco**

Applica la stessa logica per la **password Wi-Fi** in `extras.wifi.password`.

## Soft Delete — Analisi per Tabella

| Tabella | Soft Delete | Motivazione |
| :-- | :-- | :-- |
| `users` | ✅ **Sì** | Un utente cancellato ha wizard e log storici da preservare per audit. `restrictOnDelete` sulle FK garantisce l'integrità. |
| `templates` | ✅ **Sì** | Un template eliminato potrebbe essere riferito da wizard esistenti; `nullOnDelete` su `wizards.template_id` ma il template va conservato per la history. |
| `wizards` | ✅ **Sì** | Wizard con `execution_logs` associati non devono scomparire — necessario per tracciabilità e report. |
| `software_library` | ✅ **Sì** | Software "archiviato" deve rimanere visibile nei report storici. Usa `attivo=false` per nasconderlo dalla UI, soft delete per audit trail. |
| `execution_logs` | ❌ **No** | I log sono immutabili per natura — non si cancellano mai, si archiviano. Se necessario, usa un campo `archived_at`. |
| `reports` | ❌ **No** | I report sono documenti finali. `cascadeOnDelete` dal log è sufficiente; non ha senso un soft delete separato. |

## Avvocato del Diavolo — Rischi e Limiti

**Rischio 1 — Chiave APP_KEY persa.** Se la chiave Laravel (`APP_KEY` in `.env`) viene persa o ruotata, tutte le password cifrate con `Crypt::encryptString()` diventano **irrecuperabili**. Soluzione: backup separato e sicuro dell'`APP_KEY`; considera una chiave di cifratura dedicata separata dall'`APP_KEY` per i dati critici.

**Rischio 2 — JSON `configurazione` non validato.** Un JSON malformato in `configurazione` può causare eccezioni silenti. Aggiungi un Form Request con validazione JSON schema prima di salvare qualsiasi wizard.

**Rischio 3 — `LONGTEXT` per i report.** MySQL ha un limite di 4 GB per `LONGTEXT`, ma il vero collo di bottiglia è la RAM del server al momento della query. Per report molto numerosi, valuta di spostare `html_content` su filesystem (storage S3/locale) e salvare nel DB solo il path.

**Rischio 4 — `softDeletes` senza cleanup.** I record soft-deleted rimangono in tabella per sempre senza una strategia di purge. Aggiungi un `php artisan schedule:run` con un job che hard-delete i record soft-deleted più vecchi di N giorni.

**Rischio 5 — `fullText` su `software_library.nome`** richiede MySQL InnoDB (default su MySQL 8) e non funziona su MariaDB < 10.6. Verifica la versione del tuo XAMPP MySQL.[^1_2]
